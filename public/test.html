거래소 기획서

MonsterCollector의 거래소는 등록된 회원 간 아이템을 사고팔 수 있는 기능입니다.
싱글게임 기반의 특성을 고려하여, 서버는 거래소에 등록된 아이템만 관리하며 사용자의 전체 인벤토리는 서버에 저장하지 않습니다.

기본 원칙

•등록된 회원만 거래 참여 가능
•아이템은 거래소에 등록하는 시점에만 서버에 기록됨
•거래 대금은 서버에서 관리하는 예치 화폐로만 진행
•자신이 등록한 아이템은 본인이 구매 불가 (조회는 허용, 서버에서 구매 차단)

API 설계

GET /exchange
전체 active listing 반환. 판매자 본인 목록도 포함 (구매 버튼은 클라이언트에서 비활성화)
GET /exchange/my/{user_id}
본인이 등록한 listing 반환 (전체 status 포함)
GET /exchange/wallet/{user_id} 잔액 조회
user_id 유저 ID balance 현재 잔액
POST /exchange/wallet/deposit 화폐 예치
필드 설명
user_id예치 요청자
resource_type사용할 인게임 재화 종류
resource_amount소모할 수량
1. members 검증 → 2. resource 유효성 확인 → 3. 전환 비율 계산 → 4. wallets 잔액 증가
POST /exchange/wallet/withdraw 화폐 환전
필드 설명
user_id환전 요청자
amount환전할 거래소 화폐 금액
resource_type받을 인게임 재화 종류

Response 200

resource_type 지급된 재화 종류 resource_amount 지급된 수량 remaining_balance 남은 잔액

Response 200

tx_id item_id item_instance_id item_name item_grade price remaining_balance
POST /…/{id}/cancel
active 상태만 취소 허용. listings → cancelled, exchange_cache 삭제.
Response: item_id, item_instance_id, item_name, item_grade

04
거래소 방문 시 동기화 — GET /exchange/my/{user_id} 호출, sold/cancelled 항목 확인 및 UI 반영
05
아이템 회수 — POST /…/retrieve 호출, 응답 item을 로컬 인벤토리에 추가
06~08
화폐 UI — 예치(deposit) / 환전(withdraw) UI 구현, GET /…/wallet/{id}로 잔액 표시

논의 필요 사항

1 예치 가능한 인게임 재화 종류

게임 점수, 게임 재료 등 어떤 재화를 거래소 화폐로 전환할 수 있는지. GDD 골드/재화 경제 설계(미결 High)와 함께 결정 필요.

2 전환 비율

인게임 재화 N → 거래소 화폐 M 비율. 환전 시 수수료 차감 여부 및 비율. 게임 밸런스에 직접적 영향.

3 일일 예치 한도

화폐 인플레이션 방지 목적. 전환 비율과 함께 결정 권장.

4 거래 수수료

아이템 거래 시 수수료 차감 여부 및 비율. 판매자/구매자 부담 방식 결정.

5 가격 범위 제한

아이템 등록 시 최소/최대 가격. 등급별 다른 범위 적용 여부.

6 item_master 속성

type, job_class, element 외 추가 필터 속성 여부. 각 속성 허용 값 목록 확정.

7 cron 실행 주기

만료 처리 / 거래 내역 이관 / 회수 완료 이관 각각의 주기 결정. 너무 짧으면 불필요한 실행, 너무 길면 데이터 누적.
